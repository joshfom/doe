# Requirements Document

## Introduction

This document defines the requirements for OTP (One-Time Password) verification in the ORA AI assistant. Currently, the ORA AI resolves user identity by phone, email, or session and immediately grants access to personalized account data. This feature adds a verification step: when a recognized client or tenant asks about personal details (unit status, payments, account info), the AI must first authenticate them via a 6-digit OTP sent to their registered email using Microsoft Graph API. Until the OTP is verified, the AI responds only to general (non-personalized) queries. Payment-related queries always require OTP verification followed by human handoff, and sensitive financial matters (disputes, account changes) always result in a support ticket and human agent involvement. The OTP is stored in the database tied to the conversation, expires after 5 minutes, and is a 6-digit numeric code.

## Glossary

- **ORA_AI**: The autonomous AI virtual assistant system that processes user queries, identifies clients, retrieves knowledge base content, performs actions, and generates responses within defined scope boundaries
- **OTP**: A 6-digit numeric one-time password generated by the system, sent to the user's registered email, valid for a single use within a 5-minute window
- **OTP_Record**: A database entity storing the OTP code, associated conversation ID, target email address, creation timestamp, expiry timestamp, verification status, and attempt count
- **OTP_Verification_State**: The authentication state tracked on the Conversation record, indicating whether the user has been verified via OTP for the current session (one of: "not_required", "pending", "verified", "expired")
- **Personal_Query**: A user query that requests personalized account data including unit status, construction progress, payment details, lease information, or any data specific to a Client_Record or Tenant_Record
- **General_Query**: A user query about publicly available information (community details, project overviews, FAQs, general policies) that does not require identity verification
- **Payment_Query**: A user query specifically about making payments, payment status, payment disputes, or financial transactions — a subset of Personal_Query that always requires human handoff after OTP verification
- **Sensitive_Query**: A user query about payment disputes, account changes, financial corrections, or other high-risk matters that require immediate support ticket creation and human agent involvement
- **Microsoft_Graph_API**: The Azure-based email delivery service used to send OTP codes to registered email addresses, authenticated via Azure AD credentials (tenant ID, client ID, client secret)
- **Conversation**: A database entity representing a complete chat session between a user and ORA_AI, extended with OTP verification state fields
- **Client_Record**: A database entity representing an ORA client (property buyer) with personal details, contact information, and associated units
- **Tenant_Record**: A database entity representing an ORA tenant (property renter) with personal details, contact information, and lease details
- **Support_Ticket**: A ticket created in the existing ticketing system for issues requiring human agent review and resolution
- **Human_Handoff**: The process of transferring a Conversation from ORA_AI to a human agent, used after OTP verification for payment queries and always for sensitive queries

## Requirements

### Requirement 1: Personal Query Detection and OTP Gate

**User Story:** As a recognized client or tenant, I want the AI to verify my identity before sharing my personal account details, so that my data is protected from unauthorized access.

#### Acceptance Criteria

1. WHEN a recognized Client or Tenant sends a Personal_Query and the Conversation OTP_Verification_State is not "verified", THE ORA_AI SHALL withhold all personalized account data and instead respond with an authentication prompt that includes the user's first name, a description of what they asked about, an offer to send an OTP to their registered email, and a note that general questions can still be answered without verification
2. WHEN a visitor (unidentified user) sends a Personal_Query, THE ORA_AI SHALL respond that the user must first provide their phone number or email for identification, and that personal account data requires identity verification
3. WHILE the Conversation OTP_Verification_State is "verified", THE ORA_AI SHALL process Personal_Queries normally by retrieving and presenting personalized data from Client_Records, Tenant_Records, and Unit_Records
4. WHILE the Conversation OTP_Verification_State is "pending" or "not_required", THE ORA_AI SHALL respond to General_Queries normally without requiring OTP verification
5. THE ORA_AI SHALL classify incoming messages as General_Query, Personal_Query, Payment_Query, or Sensitive_Query based on intent detection keywords and context analysis

### Requirement 2: OTP Generation and Delivery

**User Story:** As a recognized client or tenant, I want to receive a one-time password on my registered email, so that I can verify my identity securely.

#### Acceptance Criteria

1. WHEN the user confirms they want to receive an OTP, THE ORA_AI SHALL generate a cryptographically random 6-digit numeric OTP code
2. WHEN an OTP is generated, THE ORA_AI SHALL create an OTP_Record in the database containing the OTP code (hashed), the Conversation ID, the target email address, the creation timestamp, an expiry timestamp set to 5 minutes from creation, a verification status of "pending", and an attempt count of 0
3. WHEN an OTP_Record is created, THE ORA_AI SHALL send the OTP code to the user's registered email address using the Microsoft_Graph_API with the configured Azure credentials (AZURE_COMMUNICATION_TENANT_ID, AZURE_COMMUNICATION_CLIENT_ID, AZURE_COMMUNICATION_CLIENT_SECRET, AZURE_COMMUNICATION_SENDER)
4. WHEN the OTP email is sent successfully, THE ORA_AI SHALL update the Conversation OTP_Verification_State to "pending" and respond to the user confirming that the OTP has been sent to their registered email (showing a masked version of the email, e.g., "a****d@example.com")
5. IF the OTP email delivery fails, THEN THE ORA_AI SHALL inform the user that the OTP could not be sent, suggest trying again, and offer to connect with a human representative as an alternative
6. WHEN an OTP is generated for a Conversation that already has a pending OTP_Record, THE ORA_AI SHALL invalidate the previous OTP_Record and create a new one (only the latest OTP is valid)

### Requirement 3: OTP Verification

**User Story:** As a client or tenant who received an OTP, I want to enter the code in the chat to verify my identity, so that I can access my personal account information.

#### Acceptance Criteria

1. WHEN a user provides a 6-digit code while the Conversation OTP_Verification_State is "pending", THE ORA_AI SHALL compare the provided code against the hashed OTP in the active OTP_Record for that Conversation
2. WHEN the provided OTP code matches and the OTP_Record has not expired, THE ORA_AI SHALL update the OTP_Verification_State to "verified", mark the OTP_Record as "used", and respond with a confirmation message (e.g., "You're verified! How can I help you with your account?")
3. WHEN the provided OTP code does not match, THE ORA_AI SHALL increment the attempt count on the OTP_Record and inform the user that the code is incorrect, showing the remaining attempts
4. WHEN the attempt count on an OTP_Record reaches 3, THE ORA_AI SHALL invalidate the OTP_Record, update the OTP_Verification_State to "expired", and inform the user that the OTP has been locked due to too many failed attempts, offering to send a new OTP
5. WHEN a user provides an OTP code after the 5-minute expiry window, THE ORA_AI SHALL inform the user that the OTP has expired and offer to send a new one
6. FOR ALL valid OTP codes, generating the OTP, hashing it, and then verifying the original code against the hash SHALL succeed (round-trip verification property)

### Requirement 4: Payment Query Handling with OTP and Human Handoff

**User Story:** As a client, I want the AI to verify my identity before discussing payments and then connect me with a human agent, so that payment matters are handled securely through official channels.

#### Acceptance Criteria

1. WHEN a user sends a Payment_Query and the Conversation OTP_Verification_State is not "verified", THE ORA_AI SHALL first trigger the OTP verification flow before proceeding with any payment-related response
2. WHEN a user sends a Payment_Query and the Conversation OTP_Verification_State is "verified", THE ORA_AI SHALL respond with payment information that references only ORA's official payment channels (as shared at time of booking) and include the warning: "No payment should be sent to any personal account — only ORA official channels"
3. WHEN a Payment_Query is answered, THE ORA_AI SHALL initiate a Human_Handoff to connect the user with a human correspondent for further payment assistance
4. THE ORA_AI SHALL include the payment safety warning in both English and Arabic depending on the Conversation language

### Requirement 5: Sensitive Query Escalation

**User Story:** As a client, I want sensitive financial matters to always involve a human agent, so that disputes and account changes are handled carefully and never autonomously by the AI.

#### Acceptance Criteria

1. WHEN a user sends a Sensitive_Query (payment disputes, account changes, financial corrections, refund requests), THE ORA_AI SHALL create a Support_Ticket in the existing ticketing system with the query details, conversation reference, and user identity information
2. WHEN a Support_Ticket is created for a Sensitive_Query, THE ORA_AI SHALL initiate a Human_Handoff and inform the user that a support ticket has been created and a human agent will follow up, providing the ticket reference number
3. THE ORA_AI SHALL NOT attempt to resolve Sensitive_Queries autonomously, regardless of the OTP_Verification_State
4. WHEN a Sensitive_Query is detected, THE ORA_AI SHALL classify the Support_Ticket with appropriate priority (high for payment disputes, medium for account changes) and include the full conversation context in the ticket description

### Requirement 6: OTP Storage and Database Schema

**User Story:** As a developer, I want OTP records stored in the database tied to conversations, so that verification state is persistent and auditable.

#### Acceptance Criteria

1. THE CMS_Platform SHALL define a Drizzle ORM schema for OTP_Records containing: id (UUID), conversation_id (foreign key to ai_conversations), otp_hash (text, the hashed OTP code), email (text, the target email address), status (enum: "pending", "used", "expired", "invalidated"), attempt_count (integer, default 0), max_attempts (integer, default 3), expires_at (timestamp), created_at (timestamp), and verified_at (timestamp, nullable)
2. THE CMS_Platform SHALL add an otp_verification_state column to the ai_conversations table with values "not_required", "pending", "verified", and "expired", defaulting to "not_required"
3. THE CMS_Platform SHALL add a database index on OTP_Record conversation_id and status for efficient lookup of active OTPs
4. THE ORA_AI SHALL store OTP codes as hashed values using a one-way hash function; raw OTP codes SHALL NOT be persisted in the database
5. WHEN a Conversation ends or is resolved, THE ORA_AI SHALL invalidate any pending OTP_Records associated with that Conversation

### Requirement 7: OTP Email Formatting

**User Story:** As a client receiving an OTP email, I want the email to be clear and professional, so that I understand the purpose and can easily find the code.

#### Acceptance Criteria

1. THE ORA_AI SHALL send OTP emails with a clear subject line indicating the purpose (e.g., "ORA — Your Verification Code")
2. THE ORA_AI SHALL format the OTP email body to include: the ORA brand name, the 6-digit OTP code displayed prominently, a note that the code expires in 5 minutes, a warning not to share the code with anyone, and contact information for support
3. THE ORA_AI SHALL send OTP emails in the language matching the Conversation language (English or Arabic)
4. THE ORA_AI SHALL use the configured AZURE_COMMUNICATION_SENDER address as the email sender

### Requirement 8: OTP Session Persistence

**User Story:** As a verified client, I want my verification to persist for the duration of my chat session, so that I do not need to re-verify for every personal question.

#### Acceptance Criteria

1. WHILE a Conversation OTP_Verification_State is "verified", THE ORA_AI SHALL allow all subsequent Personal_Queries in the same Conversation without requiring re-verification
2. WHEN a new Conversation is started (new session), THE ORA_AI SHALL require fresh OTP verification for Personal_Queries regardless of previous session verification status
3. THE ORA_AI SHALL NOT carry OTP verification state across different Conversations, even for the same identified user
