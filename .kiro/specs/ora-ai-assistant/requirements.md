# Requirements Document

## Introduction

This document defines the requirements for ORA AI — an autonomous AI virtual assistant that acts as a digital employee for ORA's real estate community platform. ORA AI identifies clients and tenants, answers personalized queries about booked units, construction updates, payments, and community details, performs autonomous actions such as booking appointments, and operates strictly within ORA's own knowledge base. The assistant integrates with the existing ORA CMS Platform (Next.js App Router, Elysia.js API, Drizzle ORM on PostgreSQL, Better Auth, TanStack Query) and extends the existing schema with client/tenant records, unit information, appointments, conversation history, and knowledge base indexing. ORA AI supports English and Arabic, provides a web chat widget on the public frontend, and includes an admin panel section for managing the knowledge base, reviewing conversations, and monitoring analytics.

## Glossary

- **ORA_AI**: The autonomous AI virtual assistant system that processes user queries, identifies clients, retrieves knowledge base content, performs actions, and generates responses within defined scope boundaries
- **Knowledge_Base**: The curated collection of ORA-owned content (blog posts, construction updates, community details, FAQs, payment guides, policy documents) indexed for semantic retrieval by ORA_AI
- **Knowledge_Document**: A single indexed content item within the Knowledge_Base, containing source text, vector embeddings, metadata (source type, category, locale, last updated), and a reference to the original content
- **RAG_Pipeline**: The Retrieval-Augmented Generation pipeline that converts a user query into a vector embedding, retrieves relevant Knowledge_Documents via similarity search, and passes them as context to the language model for response generation
- **Vector_Store**: The PostgreSQL-based vector storage (using pgvector extension) that holds embeddings for all Knowledge_Documents and supports similarity search queries
- **Embedding**: A numerical vector representation of text content generated by an embedding model, stored in the Vector_Store for semantic similarity matching
- **Client_Record**: A database entity representing an ORA client (property buyer) with personal details, contact information, associated units, payment history references, and interaction preferences
- **Tenant_Record**: A database entity representing an ORA tenant (property renter) with personal details, contact information, associated unit, lease details, and interaction preferences
- **Unit_Record**: A database entity representing a real estate unit (apartment, villa, townhouse) with details including project name, unit number, type, status, construction progress, handover timeline, and associated client or tenant
- **Conversation**: A database entity representing a complete chat session between a user and ORA_AI, containing an ordered sequence of messages, participant identification, channel, timestamps, and resolution status
- **Message**: A single exchange within a Conversation, containing the sender role (user or assistant), content text, timestamp, and optional metadata (retrieved sources, action performed)
- **Appointment**: A database entity representing a scheduled meeting between a client/tenant and an ORA representative, with date, time, type, location, status, and associated participant
- **Chat_Widget**: The embeddable web chat interface rendered on the Public_Frontend that allows visitors, clients, and tenants to interact with ORA_AI
- **Admin_AI_Panel**: The section within the existing Admin_Panel at `/ora-panel/ai/` for managing the Knowledge_Base, reviewing Conversations, configuring ORA_AI behavior, and viewing analytics
- **Scope_Boundary**: The defined set of topics and actions ORA_AI is permitted to address; queries outside this boundary trigger a graceful decline response or human handoff
- **Human_Handoff**: The process of transferring a Conversation from ORA_AI to a human agent when the assistant cannot resolve the query or the user explicitly requests human assistance
- **Content_Sync**: The automated process that indexes new or updated CMS content (blog posts, pages) into the Knowledge_Base by generating embeddings and storing them in the Vector_Store
- **Identification_Flow**: The process by which ORA_AI identifies a user as a known Client or Tenant using phone number, email, or authenticated session, and personalizes subsequent interactions

## Requirements

### Requirement 1: Client and Tenant Identification

**User Story:** As a client or tenant, I want ORA AI to recognize me from my phone number or login, so that I receive personalized responses relevant to my account.

#### Acceptance Criteria

1. WHEN a user initiates a Conversation and provides a phone number, THE ORA_AI SHALL query the Client_Record and Tenant_Record tables to determine if the phone number matches a known contact
2. WHEN a user initiates a Conversation through an authenticated session, THE ORA_AI SHALL resolve the user identity from the Auth_Session and link the Conversation to the corresponding Client_Record or Tenant_Record
3. WHEN a known Client or Tenant is identified, THE ORA_AI SHALL greet the user by first name (e.g., "Hi John, glad to connect with you today") and load their associated Unit_Records and account context into the Conversation
4. WHEN a user cannot be identified as a known Client or Tenant, THE ORA_AI SHALL proceed as a general visitor interaction without personalized account data and offer to connect the user with a human representative for account-specific queries
5. IF a user provides a phone number that matches multiple records, THEN THE ORA_AI SHALL ask the user to confirm their identity by providing additional information (e.g., email address or unit number)
6. THE ORA_AI SHALL store the resolved identity association on the Conversation record for the duration of the session

### Requirement 2: Personalized Query Handling

**User Story:** As an identified client, I want to ask ORA AI about my booked units, construction status, payment channels, and community details, so that I get accurate answers specific to my account.

#### Acceptance Criteria

1. WHEN an identified Client asks about their booked unit, THE ORA_AI SHALL retrieve the associated Unit_Records and respond with unit type, project name, unit number, and current status
2. WHEN an identified Client asks about construction progress, THE ORA_AI SHALL retrieve the latest construction update from the Unit_Record and associated Knowledge_Documents and respond with the current construction phase, percentage complete, and estimated handover timeline
3. WHEN a user asks about payment channels or payment processes, THE ORA_AI SHALL retrieve the relevant payment information from the Knowledge_Base and respond with available payment methods, bank details, and step-by-step guidance
4. WHEN a user asks about community details, THE ORA_AI SHALL retrieve community information from the Knowledge_Base including amenities, facilities, location details, and nearby services
5. WHEN an identified Client asks a pre-sale question (pricing, availability, floor plans), THE ORA_AI SHALL retrieve relevant unit availability and pricing information from the Knowledge_Base and Unit_Records
6. WHEN an identified Client asks a post-sale question (maintenance, move-in procedures, warranty), THE ORA_AI SHALL retrieve relevant post-sale documentation from the Knowledge_Base
7. IF ORA_AI cannot find sufficient information in the Knowledge_Base or Client data to answer a personalized query, THEN THE ORA_AI SHALL inform the user that the information is not available and offer to connect them with a human representative

### Requirement 3: Knowledge Base Management

**User Story:** As an administrator, I want to manage the AI knowledge base from the admin panel, so that ORA AI always has accurate and up-to-date information to draw from.

#### Acceptance Criteria

1. THE Admin_AI_Panel SHALL provide a knowledge base management interface at `/ora-panel/ai/knowledge-base` for viewing, adding, editing, and deleting Knowledge_Documents
2. WHEN an administrator adds a new Knowledge_Document, THE ORA_AI SHALL generate an Embedding for the document content and store it in the Vector_Store
3. WHEN an administrator updates an existing Knowledge_Document, THE ORA_AI SHALL regenerate the Embedding and replace the previous entry in the Vector_Store
4. WHEN a blog post or news article is published or updated in the existing CMS blog module, THE Content_Sync process SHALL automatically index the content as a Knowledge_Document in the Knowledge_Base
5. WHEN a Knowledge_Document is deleted, THE ORA_AI SHALL remove the corresponding Embedding from the Vector_Store
6. THE Admin_AI_Panel SHALL display each Knowledge_Document with its source type (manual, blog_sync, construction_update, faq, policy), category, locale, content preview, and last indexed timestamp
7. THE Admin_AI_Panel SHALL support filtering Knowledge_Documents by source type, category, and locale
8. THE ORA_AI SHALL support Knowledge_Documents in both English and Arabic locales, and THE RAG_Pipeline SHALL prefer documents matching the Conversation language when retrieving context

### Requirement 4: Knowledge Base Scope Boundaries

**User Story:** As a platform operator, I want ORA AI to only use ORA's own knowledge base and never internet information, so that responses are accurate and brand-consistent.

#### Acceptance Criteria

1. THE ORA_AI SHALL generate responses exclusively from content retrieved via the RAG_Pipeline from the Knowledge_Base and from structured data in Client_Records, Tenant_Records, and Unit_Records
2. THE ORA_AI SHALL NOT use general internet knowledge, external APIs, or information outside the Knowledge_Base to answer user queries
3. WHEN a user asks a question that falls outside the Scope_Boundary (e.g., unrelated to ORA real estate, general trivia, personal advice), THE ORA_AI SHALL respond with a polite decline message explaining that the query is outside the scope of ORA services
4. THE Admin_AI_Panel SHALL provide a scope configuration interface where administrators can define permitted topic categories and blocked topic keywords
5. WHEN a user asks about a topic that is within the Scope_Boundary but no relevant Knowledge_Documents exist, THE ORA_AI SHALL inform the user that the information is not currently available and suggest contacting a human representative
6. THE ORA_AI SHALL include source attribution in responses by referencing the Knowledge_Document title or category from which the information was retrieved

### Requirement 5: RAG Pipeline and Query Processing

**User Story:** As a developer, I want a robust retrieval-augmented generation pipeline, so that ORA AI provides accurate, contextual responses grounded in ORA's knowledge base.

#### Acceptance Criteria

1. WHEN a user sends a Message, THE RAG_Pipeline SHALL convert the message text into an Embedding using the configured embedding model
2. THE RAG_Pipeline SHALL perform a similarity search against the Vector_Store to retrieve the top-K most relevant Knowledge_Documents (configurable K, default 5)
3. THE RAG_Pipeline SHALL construct a prompt containing the retrieved Knowledge_Documents as context, the Conversation history (last N messages, configurable), the user identity context (if identified), and the user's current query
4. THE RAG_Pipeline SHALL send the constructed prompt to the configured language model and return the generated response as a new assistant Message in the Conversation
5. THE RAG_Pipeline SHALL include a relevance threshold for retrieved documents; documents below the threshold SHALL be excluded from the context to avoid injecting irrelevant information
6. WHEN the Conversation involves an identified Client or Tenant, THE RAG_Pipeline SHALL augment the context with structured data from the associated Client_Record, Tenant_Record, and Unit_Records
7. THE RAG_Pipeline SHALL log the retrieved document IDs and similarity scores on each assistant Message for debugging and analytics purposes
8. FOR ALL Knowledge_Documents, indexing the document content into the Vector_Store and then querying with the same content SHALL return that document among the top results (round-trip retrieval property)

### Requirement 6: Autonomous Actions — Appointment Booking

**User Story:** As a client, I want to book meetings and appointments through ORA AI, so that I can schedule visits and consultations without calling the office.

#### Acceptance Criteria

1. WHEN a user requests to book an appointment, THE ORA_AI SHALL collect the required information: preferred date, preferred time, appointment type (site visit, consultation, payment discussion, maintenance request), and contact details (if not already identified)
2. THE ORA_AI SHALL check available time slots against existing Appointments in the database to avoid double-booking
3. WHEN all required information is collected and a time slot is available, THE ORA_AI SHALL create an Appointment record in the database with status "confirmed" and respond with a confirmation message including date, time, type, and a reference number
4. IF the requested time slot is not available, THEN THE ORA_AI SHALL suggest the three nearest available time slots and ask the user to select one
5. WHEN an Appointment is created, THE ORA_AI SHALL record the action in the Audit_Log with the appointment details and the associated Conversation ID
6. THE Admin_AI_Panel SHALL provide an appointment management interface at `/ora-panel/ai/appointments` displaying all Appointments with filtering by date, status, and type
7. THE ORA_AI SHALL allow users to cancel or reschedule existing Appointments within the same Conversation by referencing the appointment reference number

### Requirement 7: Autonomous Actions — Record Lookup and Guidance

**User Story:** As a client, I want ORA AI to look up my records and guide me through processes, so that I get immediate answers without waiting for a human agent.

#### Acceptance Criteria

1. WHEN an identified Client asks to check their account or unit status, THE ORA_AI SHALL query the Client_Record and associated Unit_Records and present a summary including unit details, current status, and key dates
2. WHEN an identified Tenant asks about their lease, THE ORA_AI SHALL query the Tenant_Record and present lease start date, end date, rent amount, and payment due dates
3. WHEN a user asks about a payment process, THE ORA_AI SHALL retrieve the step-by-step payment guide from the Knowledge_Base and present it as a structured, numbered response
4. WHEN a user asks about construction updates for a specific project, THE ORA_AI SHALL retrieve the latest construction update Knowledge_Document for that project and present the current phase, progress percentage, and next milestone
5. IF an identified user requests information about a unit or project not associated with their record, THEN THE ORA_AI SHALL respond with general publicly available information from the Knowledge_Base without disclosing other clients' data

### Requirement 8: Conversation Management

**User Story:** As an administrator, I want to review all AI conversations, so that I can monitor quality, identify issues, and improve the knowledge base.

#### Acceptance Criteria

1. THE ORA_AI SHALL persist every Conversation and its Messages in the database with timestamps, participant identification, channel, and resolution status
2. THE Admin_AI_Panel SHALL provide a conversation review interface at `/ora-panel/ai/conversations` displaying Conversations in reverse chronological order with search by participant name, phone number, or message content
3. THE Admin_AI_Panel SHALL support filtering Conversations by status (active, resolved, handed_off), channel, date range, and whether the participant was identified
4. WHEN an administrator opens a Conversation, THE Admin_AI_Panel SHALL display the full message history with sender roles, timestamps, retrieved source references, and any actions performed
5. THE Admin_AI_Panel SHALL display a resolution status for each Conversation: "resolved" when the user's query was answered, "handed_off" when transferred to a human agent, and "abandoned" when the user left without resolution
6. THE ORA_AI SHALL automatically mark a Conversation as "resolved" when the user confirms their query is answered or after a configurable inactivity timeout

### Requirement 9: Human Handoff

**User Story:** As a user, I want to be transferred to a human agent when ORA AI cannot help me, so that I still get the support I need.

#### Acceptance Criteria

1. WHEN a user explicitly requests to speak with a human agent, THE ORA_AI SHALL initiate a Human_Handoff by updating the Conversation status to "handed_off" and notifying the admin panel
2. WHEN ORA_AI fails to provide a satisfactory answer after two consecutive attempts on the same topic (detected by repeated similar queries), THE ORA_AI SHALL proactively offer a Human_Handoff
3. WHEN a Human_Handoff is initiated, THE ORA_AI SHALL send a summary message to the user confirming that a human agent will follow up, including the Conversation reference number
4. THE Admin_AI_Panel SHALL display a notification badge for Conversations with "handed_off" status that have not been reviewed by an administrator
5. WHEN a Human_Handoff is initiated, THE ORA_AI SHALL include a handoff summary in the Conversation metadata containing the user's original query, the attempted responses, and the reason for handoff

### Requirement 10: Chat Widget

**User Story:** As a site visitor, I want a chat widget on the ORA website, so that I can interact with ORA AI from any page.

#### Acceptance Criteria

1. THE Public_Frontend SHALL render a Chat_Widget as a floating button on all public pages, positioned at the bottom-right corner of the viewport
2. WHEN a user clicks the Chat_Widget button, THE Chat_Widget SHALL expand into a chat panel displaying a welcome message and an input field for the user to type a query
3. THE Chat_Widget SHALL support real-time message exchange with ORA_AI, displaying assistant responses as they are generated
4. THE Chat_Widget SHALL persist the Conversation across page navigations within the same browser session using a session identifier stored in browser storage
5. THE Chat_Widget SHALL support both English and Arabic interfaces, matching the current locale of the Public_Frontend page, with `dir="rtl"` applied for Arabic
6. THE Chat_Widget SHALL display a typing indicator while ORA_AI is processing a response
7. WHEN the Chat_Widget is minimized, THE Chat_Widget SHALL display an unread message count badge if new assistant messages arrived while minimized
8. THE Chat_Widget SHALL be responsive and functional on both desktop and mobile viewports

### Requirement 11: Multilingual Conversation Support

**User Story:** As a user, I want to interact with ORA AI in English or Arabic, so that I can communicate in my preferred language.

#### Acceptance Criteria

1. THE ORA_AI SHALL detect the language of the user's first Message (English or Arabic) and set the Conversation language accordingly
2. WHILE a Conversation language is set to Arabic, THE ORA_AI SHALL generate all responses in Arabic
3. WHILE a Conversation language is set to English, THE ORA_AI SHALL generate all responses in English
4. WHEN a user switches language mid-conversation, THE ORA_AI SHALL detect the language change and switch the response language accordingly
5. THE RAG_Pipeline SHALL prefer Knowledge_Documents matching the Conversation language when retrieving context, falling back to the other locale if no matching documents are found

### Requirement 12: Client and Tenant Data Schema

**User Story:** As a developer, I want well-structured database tables for clients, tenants, and units, so that ORA AI can query structured account data reliably.

#### Acceptance Criteria

1. THE CMS_Platform SHALL define a Drizzle ORM schema for Client_Records containing: id, first name, last name, email, phone number, nationality, preferred language (EN/AR), associated unit IDs, notes, created timestamp, and updated timestamp
2. THE CMS_Platform SHALL define a Drizzle ORM schema for Tenant_Records containing: id, first name, last name, email, phone number, unit ID, lease start date, lease end date, rent amount, payment frequency, notes, created timestamp, and updated timestamp
3. THE CMS_Platform SHALL define a Drizzle ORM schema for Unit_Records containing: id, project name, unit number, unit type (apartment, villa, townhouse, office), floor number, area in square meters, status (available, sold, reserved, rented, under_construction), construction progress percentage, estimated handover date, client ID (nullable), tenant ID (nullable), created timestamp, and updated timestamp
4. THE CMS_Platform SHALL enforce referential integrity between Unit_Records and Client_Records, and between Unit_Records and Tenant_Records
5. THE CMS_Platform SHALL use database indexes on Client_Record phone number, Client_Record email, Tenant_Record phone number, Tenant_Record email, and Unit_Record status for efficient lookup
6. THE Admin_AI_Panel SHALL provide CRUD interfaces for managing Client_Records, Tenant_Records, and Unit_Records at `/ora-panel/ai/clients`, `/ora-panel/ai/tenants`, and `/ora-panel/ai/units` respectively

### Requirement 13: AI Configuration and Analytics

**User Story:** As an administrator, I want to configure ORA AI behavior and view usage analytics, so that I can optimize the assistant's performance and understand user needs.

#### Acceptance Criteria

1. THE Admin_AI_Panel SHALL provide a configuration interface at `/ora-panel/ai/settings` for managing ORA_AI parameters including: language model selection, embedding model selection, top-K retrieval count, relevance threshold, conversation history length, inactivity timeout, and welcome message text (EN/AR)
2. THE Admin_AI_Panel SHALL provide an analytics dashboard at `/ora-panel/ai/analytics` displaying: total conversations, conversations by status, conversations by channel, average messages per conversation, top queried topics, handoff rate, and daily conversation volume over time
3. WHEN an administrator updates an ORA_AI configuration parameter, THE ORA_AI SHALL apply the new parameter to all subsequent Conversations without requiring a system restart
4. THE Admin_AI_Panel SHALL display the Knowledge_Base health status including: total indexed documents, documents by source type, last sync timestamp, and documents with stale embeddings (content updated since last indexing)
5. THE ORA_AI SHALL log each query-response pair with latency, retrieved document count, and relevance scores for performance monitoring

### Requirement 14: Data Privacy and Security

**User Story:** As a platform operator, I want client and tenant data handled securely, so that personal information is protected and access is controlled.

#### Acceptance Criteria

1. THE API_Layer SHALL require a valid Auth_Session for all endpoints that access Client_Records, Tenant_Records, Unit_Records, Conversations, and AI configuration
2. THE ORA_AI SHALL NOT disclose one client's personal information, unit details, or payment information to another client or to unidentified users
3. WHEN an unidentified user asks about a specific client's account, THE ORA_AI SHALL decline the request and suggest the user identify themselves first
4. THE ORA_AI SHALL NOT include raw Client_Record or Tenant_Record data in the RAG_Pipeline context for unidentified users; personalized data SHALL only be injected when the user's identity is confirmed
5. THE Conversation records SHALL store participant identification separately from message content, allowing administrators to review conversations while maintaining data organization
6. THE Admin_AI_Panel SHALL log all access to Client_Records, Tenant_Records, and Conversation data in the existing Audit_Log

### Requirement 15: Content Sync with Existing CMS

**User Story:** As an administrator, I want blog posts and CMS content automatically synced to the AI knowledge base, so that ORA AI always has the latest published content.

#### Acceptance Criteria

1. WHEN a blog post is published in the existing blog module, THE Content_Sync process SHALL automatically create a corresponding Knowledge_Document with the post title, content, category, tags, locale, and publication date
2. WHEN a published blog post is updated, THE Content_Sync process SHALL update the corresponding Knowledge_Document and regenerate its Embedding in the Vector_Store
3. WHEN a blog post is trashed or unpublished, THE Content_Sync process SHALL remove the corresponding Knowledge_Document and its Embedding from the Vector_Store
4. THE Content_Sync process SHALL index blog post content by extracting plain text from the Tiptap JSON content structure
5. THE Admin_AI_Panel SHALL display the sync status for each blog post in the knowledge base view, indicating whether the Knowledge_Document is up-to-date, pending re-index, or missing
6. THE Content_Sync process SHALL support manual re-indexing of all blog content via an admin action in the Admin_AI_Panel

### Requirement 16: API Endpoints for ORA AI

**User Story:** As a developer, I want dedicated API endpoints for the AI assistant, so that the chat widget and admin panel can interact with ORA AI through a consistent interface.

#### Acceptance Criteria

1. THE API_Layer SHALL expose a POST `/api/ai/chat` endpoint that accepts a message text, conversation ID (optional, for continuing conversations), and user identification (optional) and returns the assistant's response message with metadata
2. THE API_Layer SHALL expose a GET `/api/ai/conversations` endpoint (auth required) for listing Conversations with pagination, filtering, and search
3. THE API_Layer SHALL expose a GET `/api/ai/conversations/:id` endpoint (auth required) for retrieving a full Conversation with all Messages
4. THE API_Layer SHALL expose CRUD endpoints for Knowledge_Documents at `/api/ai/knowledge-base` (auth required)
5. THE API_Layer SHALL expose CRUD endpoints for Client_Records at `/api/ai/clients`, Tenant_Records at `/api/ai/tenants`, and Unit_Records at `/api/ai/units` (auth required)
6. THE API_Layer SHALL expose CRUD endpoints for Appointments at `/api/ai/appointments` (auth required)
7. THE API_Layer SHALL expose a GET `/api/ai/analytics` endpoint (auth required) returning aggregated conversation and usage statistics
8. THE API_Layer SHALL validate all incoming request bodies against typed schemas and return descriptive error responses for invalid input
